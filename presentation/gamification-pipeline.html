Problem Statement & Requirements

User Features

  • Daily streaks — consecutive-day tracking
  • Bestie Points (BP) — reward interactions
  • Leaderboards — top streaks by region
  • Analytics — engagement trends, cohort analysis

Engineering Constraints

  • Scale: millions of DAU, spiky traffic
  • Latency: sub-10ms reads for streak display
  • Global TZ: streaks depend on user's local calendar day
  • Correctness: no lost/inflated streaks
  • Cost: pay-per-use — no idle provisioning
Two sides to this coin: on the left, what users see — streaks, points, leaderboards. On the right, the hard engineering constraints. Timezone handling is surprisingly critical — a user at 23:30 in São Paulo is on a different calendar day than UTC would suggest. We need correctness without sacrificing performance.

Architecture Overview

Real-time path Batch / analytics path
This is the full system. Green path is real-time: mobile app → API Gateway → Lambda ingestion. The ingestion Lambda checks Redis for rate limiting, then writes to DynamoDB (three tables: BPEvents, DailyActivity, UserStreaks). DynamoDB Streams picks up the DailyActivity change and triggers a second Lambda — the streak processor — which calculates the streak and does two things: updates UserStreaks in DynamoDB with the new streak count, and does a write-through to Redis so the next app open is sub-10ms. Blue path is batch: Streams fan out to Kinesis Firehose → S3 raw → Glue ETL → S3 curated → Redshift for analytics. Two completely independent paths, same source of truth.

Event Ingestion — 5 Steps

  1. Validate — schema, required fields, timestamp sanity
  2. Deduplicate — check IdempotencyIndex GSI
  3. Normalize — UTC conversion + user's local date
  4. Write — conditional PutItem to BPEvents
  5. Award — atomic increment on total_bestie_points

Five steps on every incoming event. The key insight is the 3 writes are NOT atomic — BPEvents, DailyActivity, UserStreaks are separate tables. We chose to accept this trade-off rather than pay the latency/cost of TransactWriteItems. The nightly reconciliation job catches any drift.

Deduplication — 3-Layer Defense

LayerMechanismCatchesConsistency
1Client idempotency keyMissing key → reject 400N/A
2GSI lookup on BPEventsKnown duplicates (eventual)Eventually consistent
3Conditional PutItemRace-condition duplicatesStrongly consistent
Three layers because no single layer catches everything. Layer 1 is just key presence — rejects malformed requests. Layer 2 queries the GSI but it's eventually consistent, so two Lambda invocations could both pass the check. Layer 3 is the safety net — conditional PutItem with attribute_not_exists is strongly consistent. Bug fix #1 was adding Layer 2 — the original code relied only on Layer 3, which doesn't return the cached response for known duplicates.

Timezone Handling

The Problem

User in São Paulo at 23:30 local → UTC says March 16, but it's still March 15 for them.

The Solution: Noon Anchor

def get_previous_date(date_str: str, timezone: str) -> str:
    """Return the calendar date before the given date
    in the user's timezone. Noon anchor avoids DST edge
    cases at midnight boundaries."""
    tz = ZoneInfo(timezone)
    local_date = datetime.strptime(date_str, "%Y-%m-%d") \
        .replace(hour=12, tzinfo=tz)
    previous = local_date - timedelta(days=1)
    return previous.strftime("%Y-%m-%d")
This slide shows why timezone handling is a first-class concern. A naive UTC comparison would break streaks for anyone not on GMT. The noon anchor technique is key — by anchoring at noon instead of midnight, we avoid DST transition issues where subtracting 24 hours from midnight might land on the same calendar day. Bug fix #3 was making get_previous_date actually use the timezone parameter.

Streak Processor

  • Optimistic locking: version attribute prevents concurrent overwrites
  • Idempotent: re-processing same event produces same result
  • Cache invalidation: write-through to Redis after every update
The streak processor Lambda is triggered by DynamoDB Streams. It reads the user's current streak, compares dates in the user's local timezone, and either increments, resets, or skips. Conditional writes with version numbers prevent race conditions — if two Lambdas process events for the same user simultaneously, one will fail and retry with fresh state. The entire flow is idempotent.

Why Write-Through?

  • Streak data must be immediately consistent
  • User updates streak → sees new count instantly
  • Write-behind would introduce staleness window
  • Cache failure is non-fatal — DynamoDB is source of truth
ElastiCache Redis sits between Lambda and DynamoDB. Write-through means every DynamoDB update also writes to Redis. This gives us immediate consistency for the user — they see their new streak right away. Five types of keys with different TTLs based on their update patterns. Rate limiting and processed-event keys enable dedup and abuse detection at the stream processing layer.

Batch ETL Pipeline

Glue ETL Steps

  1. Dedup by idempotency_key (PySpark window functions)
  2. Schema enforcement & type casting
  3. Enrich with user dimension data (country, segment)
  4. Compute daily aggregates per user
  5. Write to curated S3 (Parquet, partitioned by date)
  6. COPY into Redshift
The batch path runs hourly via Step Functions. DynamoDB Streams fan out to Kinesis Firehose, which buffers to S3 in Parquet format, partitioned by date/hour. Glue ETL deduplicates using PySpark window functions — partition by idempotency_key, order by created_at, keep row_number == 1. Same dedup approach as the real-time path but in batch. In the POC, we simulate this with pandas DataFrames.

Incremental Load Pattern

  • COPY new batch into stg_bp_events (staging table)
  • DELETE matching rows from fact, INSERT from staging, TRUNCATE staging
  • Upsert behavior — idempotent, safe to re-run
Redshift uses a star schema. DISTKEY on user_id co-locates all of a user's events on the same node for fast joins. SORTKEY on event_date enables zone-map pruning for date-range queries. The staging plus merge pattern simulates Redshift COPY — bulk load into staging, delete-insert into fact, truncate staging. This is idempotent and safe to re-run. In the POC, we simulate this with SQLite in-memory.

Nightly Reconciliation

Streaks are a trust contract. Lost streaks destroy engagement, inflated streaks destroy meaning. Six anomaly detection rules cover rate abuse, timestamp manipulation, timezone gaming, and statistical outliers. The nightly reconciliation job compares DynamoDB state against DailyActivity records and BPEvents sums — auto-fixes minor drift, alerts on-call for large discrepancies.

Engineering Trade-offs

DecisionChoseOverRationale
DatabaseDynamoDBAuroraHTTP-based (no conn pool), auto-scale, conditional writes, Streams for CDC
CacheWrite-through RedisDAX / write-behindImmediate consistency; Redis supports computed aggregates + rate limiting
AtomicityEventual (3 writes)TransactWriteItemsAvoid 4x latency/cost; nightly reconciliation catches drift
ConsistencyOptimistic lockingPessimistic locksLow contention (1 user = 1 partition); conditional writes + retry
Dedup3-layer defenseSingle layerEach layer catches different failure modes (eventual vs strong)
TimezoneServer-side conversionTrust client datePrevents gaming; noon anchor handles DST safely
Six key decisions and their trade-offs. The most interesting is the atomicity trade-off — DynamoDB doesn't support efficient cross-table transactions at scale. Rather than pay 4x the cost and latency of TransactWriteItems, we accept eventual consistency and let the nightly reconciliation job fix any drift. In practice, drift is extremely rare and always small.
